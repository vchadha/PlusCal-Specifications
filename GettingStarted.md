# TLA+ / PlusCal: Installation & Getting Started Guide

TLA+ is a formal specification language for describing and verifying the correctness of systems —
particularly concurrent and distributed ones. **PlusCal** is a higher-level algorithmic language
that compiles down to TLA+. Most people write PlusCal and let the tools translate it; you only
need to read raw TLA+ when debugging or writing advanced properties.

The toolchain revolves around **TLC**, the model checker. TLC exhaustively explores every possible
state your system can reach and checks that your invariants hold in all of them. When it finds a
violation it prints an exact step-by-step error trace showing you how you got there.

> **Prerequisite for all three sections:** Java 21+. See [Installing Java](#0-prerequisite-java-21-via-sdkman)
> below before continuing.

## 0. Prerequisite: Java 21 via SDKMAN

All TLA+ tooling runs on the JVM. The recommended way to install Java on macOS and Linux is
[SDKMAN](https://sdkman.io), which lets you manage multiple Java versions without touching system
paths. Windows users can use SDKMAN inside WSL, or download OpenJDK 21 directly from
[Microsoft's OpenJDK page](https://learn.microsoft.com/en-us/java/openjdk/download#openjdk-21).

### Install SDKMAN

```bash
curl -s "https://get.sdkman.io" | bash
source "$HOME/.sdkman/bin/sdkman-init.sh"
```

### Install OpenJDK 21

```bash
sdk install java 21.0.3-ms   # Microsoft OpenJDK 21 — or pick any 21.x distribution
java -version                 # should print: openjdk version "21.x.x" ...
```

> **Tip:** Run `sdk list java` to see all available distributions and versions. Any `21.x`
> or later build works fine.

## The Toy Example: `Counter.tla`

Throughout this guide we use a single simple specification: a counter that starts at `0`,
increments by `1` on each step, and must never exceed `5`. This is trivially small, but it
exercises the full toolchain — PlusCal translation, model configuration, invariant checking,
and error traces.

### `Counter.tla`

```tla
---- MODULE Counter ----
EXTENDS Integers

(*--algorithm Counter
variables
  count = 0;

begin
  Loop:
    while TRUE do
      count := count + 1;
    end while;
end algorithm; *)

\* BEGIN TRANSLATION - the following section is auto-generated by the PlusCal translator
\* (leave this blank for now — the tools will fill it in)
\* END TRANSLATION

CounterInvariant == count <= 5

====
```

A few things to notice:

- The module name (`Counter`) **must match the filename** (`Counter.tla`), case-sensitively.
- The module is delimited by `---- MODULE Counter ----` at the top and `====` at the bottom.
- The PlusCal algorithm lives inside a `(*-- ... *)` block comment, so the file is still valid
  TLA+ before translation.
- `CounterInvariant` is a TLA+ operator defined *outside* the algorithm. The model checker will
  verify it holds in every reachable state.
- The `\* BEGIN TRANSLATION` / `\* END TRANSLATION` markers are where the translator will inject
  the generated TLA+. **Do not edit between these markers manually.**

### `Counter.cfg`

The model checker needs a configuration file describing what to check. Create `Counter.cfg`
in the same directory:

```
SPECIFICATION Spec

INVARIANT CounterInvariant
```

- `SPECIFICATION Spec` names the temporal formula that defines the behaviour (the translator
  generates an operator called `Spec` in the `BEGIN TRANSLATION` block).
- `INVARIANT CounterInvariant` tells TLC to verify the operator we defined.

> **Expected result:** TLC will find a violation. The error trace will show `count` stepping from
> `0` → `1` → … → `6`, at which point `CounterInvariant` (`count <= 5`) is false. This is the
> intended outcome for a learning exercise — you can verify the toolchain is working correctly.

## Part 1 — VSCode Extension (Recommended)

The [official TLA+ Foundation VSCode extension](https://marketplace.visualstudio.com/items?itemName=tlaplus.vscode-ide)
(`tlaplus.vscode-ide`) is the fastest path to a productive TLA+ environment. It wraps the
same `tla2tools.jar` used by the Toolbox and CLI, but surfaces everything through VS Code's
UI and command palette.

### 1.1 Install the Extension

1. Open VS Code.
2. Go to the **Extensions** panel (`Ctrl+Shift+X` / `Cmd+Shift+X`).
3. Search for **TLA+** and install the one published by **TLA+ Foundation** (identifier:
   `tlaplus.vscode-ide`). Make sure you're not installing the older `alygin.vscode-tlaplus`
   package — the Foundation extension is the actively maintained successor.

> Alternatively, open Quick Open (`Ctrl+P`) and paste:
> `ext install tlaplus.vscode-ide`

### 1.2 Workflow: Translate, Configure, Check

**Step 1 — Create your spec file.**

Create `Counter.tla` with the content shown in the [toy example](#the-toy-example-countertla)
above, and create `Counter.cfg` in the same directory.

**Step 2 — Parse and translate PlusCal.**

Open `Counter.tla` in VS Code and open the Command Palette (`Ctrl+Shift+P` / `Cmd+Shift+P`).
Run:

```
TLA+: Parse module
```

This does two things: it runs the PlusCal translator (filling in the `BEGIN TRANSLATION` block)
and then runs the SANY parser to check TLA+ syntax. Errors appear in VS Code's **Problems**
panel. On success the translation is written directly into `Counter.tla`.

**Step 3 — Run the model checker.**

With `Counter.tla` open, run from the Command Palette:

```
TLA+: Check model
```

The extension looks for `Counter.cfg` automatically (same name, same directory). TLC output
streams into a dedicated panel. Because our counter is unbounded, you will see an invariant
violation. Click the error trace to expand it and walk through the states.

> To check against a different `.cfg` file, use **`TLA+: Check model with non-default config...`**
> and pick your file from the dialog.

### 1.3 Parse on Save (Recommended)

Triggering "Parse module" manually gets tedious. You can make it run every time you save a
`.tla` file by adding this to your VS Code `settings.json`
(`Ctrl+Shift+P` → *Preferences: Open User Settings (JSON)*):

```json
"[tlaplus]": {
    "editor.codeActionsOnSave": {
        "source": true
    }
}
```

From now on, saving any `.tla` file automatically translates PlusCal and parses the module.

### 1.4 Generate a State-Space Graph (dot / Graphviz)

TLC can dump its explored state space as a Graphviz `.dot` file — useful for visualising small
specs. Add extra TLC flags via the extension setting **`TLC Model Checker: Options`**
(VS Code Settings → Extensions → TLA+):

```
-dump dot,colorize,actionlabels states.dot
```

- `colorize` colours edges by the action that caused the transition.
- `actionlabels` labels each edge with the action name.
- The output file `states.dot` is written relative to the spec directory.

Render it with Graphviz:

```bash
dot -Tsvg states.dot -o states.svg
```

> **Note:** TLC does *not* append the `.dot` extension automatically. Always include it in the
> filename you pass.

### 1.5 Export to LaTeX / PDF

The extension can render a spec as a typeset document using Tla2Tex.

- **`TLA+: Export module to LaTeX`** — generates `.tex` and `.dvi` files.
- **`TLA+: Export module to PDF`** — generates a PDF.

PDF generation requires a LaTeX distribution (`pdflatex`) to be on your `PATH`. On macOS:
`brew install --cask mactex`. On Ubuntu/Debian: `sudo apt install texlive-full`.

You can customise the PDF command in settings via **`PDF: Convert Command`**. Line numbers,
comment shading, and PlusCal shading are also individually toggleable under the PDF settings.

### 1.6 Key Commands Reference

| Command | What it does |
|---|---|
| `TLA+: Parse module` | Translate PlusCal → TLA+, then parse with SANY |
| `TLA+: Check model` | Run TLC with the default `.cfg` file |
| `TLA+: Check model with non-default config...` | Run TLC with a chosen `.cfg` file |
| `TLA+: Run last model check again` | Re-run TLC without switching files |
| `TLA+: Stop model checking` | Kill the running TLC process |
| `TLA+: Evaluate constant expression` | Evaluate a selected TLA+ expression |
| `TLA+: Visualize TLC output` | Open the graphical results view |
| `TLA+: Export module to LaTeX` | Generate `.tex` / `.dvi` |
| `TLA+: Export module to PDF` | Generate a PDF document |

> **Keyboard shortcuts:** The extension ships with no default bindings to avoid conflicts.
> Open the Keyboard Shortcuts editor (`Ctrl+K Ctrl+S`), type `TLA+` to filter, and bind
> whichever commands you use most often.

## Part 2 — TLA+ Toolbox (Legacy IDE)

The TLA+ Toolbox is the original dedicated IDE for TLA+, built on Eclipse. It is no longer
actively developed, but it remains fully functional and is well-documented. If you prefer a
self-contained GUI without VS Code, or need to follow older tutorials, the Toolbox works well.

### 2.1 Install the Toolbox

Download the latest release for your platform from the
[TLA+ GitHub releases page](https://github.com/tlaplus/tlaplus/releases/tag/v1.8.0).
Java must be installed and on your `PATH` (see [Section 0](#0-prerequisite-java-21-via-sdkman)).

Unzip the archive and run the `toolbox` (macOS/Linux) or `toolbox.exe` (Windows) binary.

### 2.2 Create a Specification

1. Go to **File → Open Spec → Add New Spec**.
2. Browse to the directory where you saved `Counter.tla` and select it (or create a new file).
3. The editor opens with a blank module skeleton. Replace it with the contents of
   `Counter.tla` from the [toy example](#the-toy-example-countertla).

The Toolbox enforces the same rule as the CLI: the module name in the first line must match
the filename exactly.

### 2.3 Translate PlusCal

Go to **File → Translate PlusCal Algorithm** (or press `Ctrl+T` on Windows/Linux, `Cmd+T` on
macOS). The Toolbox runs the translator and injects the generated TLA+ between the
`BEGIN TRANSLATION` / `END TRANSLATION` markers, exactly as the VSCode extension does.

### 2.4 Create a Model and Set Invariants

1. Go to **TLC Model Checker → New Model** and give it a name (e.g. `Model_1`).
2. The model editor opens. Under **"What is the behavior spec?"**, confirm it is set to
   **Temporal Formula** with value `Spec`. If it shows something else, check that the
   translation ran and that there is only one `====` in the file.
3. Scroll to the **Invariants** section and click **Add**. Type `CounterInvariant`.
4. Press **F11** (or the green Run button) to start TLC.

### 2.5 Read the Error Trace

When TLC finds the invariant violation, a red error panel appears on the right. This is the
**error trace** — an ordered list of states, each showing the value of every variable at that
step. You will see `count` increasing from `0` up through `6`, with the final state flagged as
the violating one.

Click any state row to expand it and inspect all variable values. The Toolbox also has a
**Trace Explorer** that lets you add derived expressions (e.g. `count + 1`) to the trace view
for richer debugging.

### 2.6 The Scratch File Pattern

When experimenting with TLA+ operators, it is useful to have a scratch spec that you can
evaluate expressions in without building a full model. Create `scratch.tla`:

```tla
---- MODULE scratch ----
EXTENDS Integers, TLC, Sequences

Eval == 1 + 1
====
```

Create a model for it, set **"What is the behavior spec?"** to **No behavior spec**, and on the
**Model Checking Results** tab put `Eval` in the **Evaluate Constant Expression** box. Running
the model prints the value of `Eval`. Change `Eval` to explore any operator you like.

### 2.7 Generating dot / LaTeX / PDF from the Toolbox

The Toolbox does not have a built-in dot export button, but you can pass additional TLC flags
via **TLC Options → TLC command line parameters** in the model editor. Add:

```
-dump dot,colorize,actionlabels states.dot
```

For PDF/LaTeX export, go to **File → Produce PDF Version of Module**. This invokes Tla2Tex and
requires `pdflatex` on your `PATH`.

## Part 3 — CLI (`tla2tools.jar`)

The CLI is ideal for scripting, CI pipelines, and automation. Everything the Toolbox and VSCode
extension do under the hood ultimately calls `tla2tools.jar`.

### 3.1 Get `tla2tools.jar`

Download the latest release from the
[TLA+ GitHub releases page](https://github.com/tlaplus/tlaplus/releases). Look for
`tla2tools.jar` in the assets. Alternatively, if you have the Toolbox installed, `tla2tools.jar`
lives in its base installation directory.

Save it somewhere on your `PATH`, or just alongside your spec files.

### 3.2 Translate PlusCal

```bash
java -cp tla2tools.jar pcal.trans Counter.tla
```

This translates the PlusCal inside `Counter.tla` and writes the result back into the same file
between the `BEGIN TRANSLATION` / `END TRANSLATION` markers. It also:

- Creates `Counter.old` as a backup of the pre-translation file.
- Creates `Counter.cfg` if one does not already exist.

To skip creating/overwriting `Counter.cfg`, add `-nocfg`:

```bash
java -cp tla2tools.jar pcal.trans -nocfg Counter.tla
```

There is no flag to suppress `Counter.old`. If that bothers you, add a `find . -name "*.old" -delete`
to your build script.

### 3.3 Run TLC

```bash
java -jar tla2tools.jar -config Counter.cfg -workers auto Counter.tla
```

- `-config Counter.cfg` — points TLC at the configuration file. If omitted, TLC looks for
  `Counter.cfg` automatically.
- `-workers auto` — use one worker thread per CPU core. **This is very important.** Without it,
  TLC defaults to a single thread and is much slower on large models.

You should see an invariant violation with a trace showing `count` incrementing past `5`.

### 3.4 The `.cfg` File Format

The config file is a small DSL. A representative example:

```
SPECIFICATION Spec

INVARIANT CounterInvariant, TypeInvariant

PROPERTY LivenessProperty

CONSTANT
  MaxCount = 5
  NodeId = {n1, n2, n3}

CHECK_DEADLOCK FALSE
```

| Keyword | Purpose |
|---|---|
| `SPECIFICATION` | The temporal formula operator (usually `Spec`) |
| `INVARIANT` | State predicate(s) to check in every state; comma-separated |
| `PROPERTY` | Temporal property (liveness) to check |
| `CONSTANT name = value` | Assign a value to a `CONSTANT` in the spec |
| `CONSTANT name = name` | Assign a *model value* (opaque atom) |
| `CONSTRAINT` | State predicate that prunes the state space (use carefully) |
| `CHECK_DEADLOCK FALSE` | Disable the default deadlock check |
| `ALIAS AliasOp` | Replace error trace output with a custom record expression |

> **Gotcha:** Config files cannot use imports. Because negative numbers technically require the
> `Integers` module, you cannot assign a negative number as a constant in the config. Work around
> this with a `CONSTANT` in the spec that is computed from a non-negative value.

### 3.5 Generate a State-Space Graph (dot)

```bash
java -jar tla2tools.jar \
  -config Counter.cfg \
  -workers auto \
  -dump dot,colorize,actionlabels states.dot \
  Counter.tla
```

Render the graph:

```bash
dot -Tsvg states.dot -o states.svg
dot -Tpng states.dot -o states.png
```

> If your spec has a liveness property, TLC also writes a `states_liveness.dot` file alongside
> it. This is an internal representation and can be ignored.

### 3.6 Export to LaTeX / PDF

```bash
# Generate .tex and .dvi
java -cp tla2tools.jar tla2tex.TLA -latexCommand pdflatex Counter.tla

# Or generate PDF directly
java -cp tla2tools.jar tla2tex.TLA -shade -number Counter.tla
```

`pdflatex` must be on your `PATH`.

### 3.7 Key TLC Flags

| Flag | Description |
|---|---|
| `-workers auto` | Use one thread per CPU core (highly recommended) |
| `-config file.cfg` | Specify the config file (defaults to `<spec>.cfg`) |
| `-continue` | Keep checking after the first violation; dumps all violations |
| `-dump dot file` | Export state graph (no extension appended automatically) |
| `-dump dot,colorize,actionlabels file` | State graph with coloured, labelled edges |
| `-metadir dir` | Write TLC's state-space cache to `dir` instead of alongside the spec |
| `-fpmem 0.5` | Fraction of system RAM to use for fingerprint storage (default: `0.25`) |
| `-noGenerateSpecTE` | Suppress writing the error-trace spec file on violation |
| `-help` | Print all available flags (use `-help`, not `-h`) |
